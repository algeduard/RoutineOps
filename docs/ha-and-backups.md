# Высокая доступность (HA) и авто-бэкапы

Как запускать RoutineOps на нескольких узлах за балансировщиком и как настроить
регулярные бэкапы БД с восстановлением. Документ честно разделяет **что уже
готово** и **что follow-up** — «полного» авто-failover из коробки нет, и это
осознанная граница.

> Смежные документы: полная процедура аварийного восстановления с нуля и список
> критичных активов — [operations.md](operations.md); установка и переменные —
> [install.md](install.md).

---

## Коротко: что готово, что нет

| Возможность | Статус |
|---|---|
| Сервер stateless (сессии в JWT-cookie, состояние в общих Postgres/Redis) | **готово** |
| Несколько узлов сервера за LB без sticky-сессий | **готово** |
| Readiness-проба `/readyz` для LB/оркестратора (реальный пинг БД и Redis) | **готово** |
| Liveness-проба `/healthz` (процесс жив) | **готово** (без изменений) |
| Авто-бэкапы Postgres с ротацией (compose-сервис `backup` / `scripts/backup.sh`) | **готово** |
| Восстановление из дампа (`scripts/restore.sh`) | **готово** |
| Репликация Postgres/Redis | **внешняя** (стандартная репликация или managed — см. ниже) |
| Авто-failover БД, leader-election для фоновых тикеров | **follow-up** (см. [«Границы»](#границы-и-follow-up)) |

## 1. Почему сервер можно масштабировать горизонтально

Сервер **stateless**: между запросами он не держит в памяти ничего, чего нельзя
было бы восстановить из общего хранилища.

- **Сессии админов** — в подписанном JWT (HS256), который лежит в httpOnly-cookie
  у браузера. Сервер их не хранит: любой узел с тем же `JWT_SECRET` проверит любую
  cookie. Отсюда — **sticky-сессии на LB НЕ нужны**.
- **Данные** (устройства, политики, скрипты, аудит, пользователи, релизы) — в общей
  **PostgreSQL**.
- **Очередь доставки задач** (asynq) — в общем **Redis**.
- **Живые gRPC-стримы агентов** (`Connect`) — единственное «локальное» состояние: агент
  держит постоянное соединение с ТЕМ узлом, к которому подключился (in-memory `registry`).
  Это не мешает масштабированию: доставка задач идёт через Redis-очередь, а
  реконсайлер повторно ставит pending-задачи в очередь, и их подхватит воркер на узле,
  где агент реально подключён (см. [раздел 4](#4-фоновые-тикеры-при-нескольких-узлах)).

Требования, чтобы N узлов вели себя одинаково:

1. **Один и тот же `JWT_SECRET`** на всех узлах (иначе cookie, выданная одним узлом, не
   пройдёт на другом → произвольные разлогины).
2. **Один и тот же mTLS-материал** (`certs/` — CA, серверный cert/key): агенты доверяют
   CA, а не конкретному узлу; серверный cert должен покрывать адрес LB в SAN.
3. **Общие** `DATABASE_DSN` и `REDIS_ADDR` (указывают на общие Postgres/Redis, а не на
   локальные к узлу).
4. Одинаковый `ROUTINEOPS_MFA_ENC_KEY` (иначе TOTP-секреты, зашифрованные одним узлом, не
   расшифрует другой).

## 2. Несколько узлов за балансировщиком

Схема: `L4/L7 LB → [server-1, server-2, …] → общие Postgres + Redis`.

- **HTTPS (REST/веб, :8081 внутри контейнера, :443 через nginx)** — балансируйте как
  обычный stateless-HTTP, round-robin. Sticky-сессии не включайте.
- **gRPC (:50051)** — агент держит долгоживущий `Connect`-стрим. L4-балансировки (TCP)
  достаточно: агент осядет на одном узле и будет получать задачи через общий Redis.
  При падении узла агент переподключится (`Restart=always`/`keepalive`) и попадёт на живой.

### Readiness через `/readyz`

Настройте health-check LB/оркестратора на **`GET /readyz`** (а не на `/healthz`):

- **`/readyz`** — *readiness*: реально пингует зависимости. `200` — узел готов принимать
  трафик; `503` — не готов (в теле `{"status":"not ready","checks":{"database":"down",…}}`
  указано, какой компонент лёг). Пробы под общим коротким таймаутом (2с), Redis-проба
  выполняется, только если Redis сконфигурирован. Смысл — **не слать трафик на узел с
  лёгшей БД/Redis**.
- **`/healthz`** — *liveness*: `200 {"status":"ok"}`, только «процесс жив». Для «перезапусти
  зависший контейнер», НЕ для «выведи узел из ротации». Поведение не изменилось.

> Сырой текст ошибки `/readyz` в тело НЕ кладёт (эндпоинт публичный — не раскрываем
> внутренние адреса/DSN); детали пишутся в лог сервера (`readyz: … ping failed`).

Пример (Kubernetes-стиль):

```yaml
readinessProbe:
  httpGet: { path: /readyz, port: 8081, scheme: HTTPS }
  periodSeconds: 10
livenessProbe:
  httpGet: { path: /healthz, port: 8081, scheme: HTTPS }
  periodSeconds: 20
```

> В штатном `docker-compose.prod.yml` REST слушает `127.0.0.1:8081` и наружу его
> проксирует nginx (`/healthz` и `/readyz` — публичные, без авторизации). Для внешнего
> LB пробуйте `/readyz` через nginx на 443 либо откройте узловой health-порт под LB.

## 3. Что реплицировать

HA приложения (несколько stateless-узлов) **не** делает отказоустойчивыми сами
Postgres и Redis — это отдельный слой, и он остаётся за вами:

- **PostgreSQL** — единственный источник истины. Для отказоустойчивости используйте
  **стандартную потоковую репликацию** (primary + hot standby, ручной/автоматический
  promote) либо **managed-Postgres** (RDS/Cloud SQL/и т.п. с встроенным HA). Приложение
  подключается по одному `DATABASE_DSN` — заведите его на writer-endpoint (или на
  локальный pgbouncer/HAProxy, который сам следит за primary). Авто-failover самой БД —
  забота этого слоя, не сервера.
- **Redis** — держит очередь asynq. Потеря Redis не теряет задачи БЕЗВОЗВРАТНО: строки
  задач persistent в Postgres (`status='pending'`), и реконсайлер (раз в минуту) заново
  ставит их в очередь. Тем не менее для непрерывности используйте Redis с персистентностью
  (AOF) и, при желании, Sentinel/managed-Redis. `REDIS_ADDR` — один адрес (writer/Sentinel-прокси).

Обязательный бэкап-набор (дамп БД + `certs/` + `release_ed25519.pem` + `.env.prod`) и
почему каждый критичен — [operations.md, раздел 1](operations.md).

## 4. Фоновые тикеры при нескольких узлах

`cmd/server/main.go` гоняет фоновые тикеры (чистка просроченных токенов/MFA-challenge/
SSO-flow, `CleanupOldData`, `ExpireStaleAdminRequests`, реконсайлер pending-задач,
детектор `agent_unreachable`). **При N узлах каждый тикер выполняется на КАЖДОМ узле
независимо** — сейчас leader-election нет. Разбор по безопасности:

- **Чистки (`DELETE … WHERE expired`)** — `CleanupExpiredRevokedTokens`,
  `DeleteExpiredMFAChallenges`, `DeleteExpiredSSOFlows`, `CleanupOldData`,
  `FailStaleAckedTasks`, `ExpireStaleAdminRequests`: **идемпотентны**. Два узла,
  удаляющие/обновляющие одни и те же истёкшие строки одновременно, дают тот же
  результат (второй просто затронет 0 строк). **Безопасно.**
- **Реконсайлер pending-задач** (ре-энкью в очередь): постановка дедуплицируется asynq
  по `TaskID` (`worker.Enqueue` → `asynq.TaskID`, повтор = `ErrTaskIDConflict` → no-op).
  N узлов, ре-энкьюящих одну и ту же задачу, приведут к **одной** доставке. Издержки —
  лишь повторный дешёвый `SELECT` pending-задач на каждом узле раз в минуту. **Безопасно**
  (дубля доставки нет).
- **Детектор `agent_unreachable`** — ⚠️ **требует внимания.** Он вставляет алерт через
  `INSERT … SELECT … WHERE NOT EXISTS(...)`. Гуард `NOT EXISTS` **не закрывает гонку двух
  ОДНОВРЕМЕННЫХ одинаковых INSERT-ов** (это прямо отмечено в комментарии `CreateAlert` в
  `storage/postgres.go`): два узла, отработавшие детектор в одну секунду, оба пройдут
  проверку и создадут **дубль** алерта (и, при настроенном Telegram, две нотификации).
  Дубль безвреден для данных (лишняя строка алерта), но шумит. Это осознанный **follow-up**.

### Практическая рекомендация до появления leader-election

Пока leader-election нет, простой и безопасный способ убрать дубли алертов —
**запускать фоновые тикеры только на одном выделенном узле**. Кода для флага пока нет
(follow-up ниже), поэтому на практике:

- держите один узел «первичным» (он же обрабатывает и фон), остальные — только под
  HTTP/gRPC-трафиком; **или**
- смиритесь с редким дублем `agent_unreachable` (данные целы), пока не приедет
  leader-election.

Чистки и реконсайлер на всех узлах оставлять безопасно (см. выше).

## 5. Авто-бэкапы Postgres

Дампы снимает **compose-сервис `backup`** (`docker-compose.prod.yml`) — контейнер
`postgres:16-alpine` ради `pg_dump`, запускающий `scripts/backup.sh` в цикле: первый
дамп сразу при старте стека, затем раз в `BACKUP_INTERVAL_SECONDS` (по умолчанию сутки).

- Формат — `pg_dump -Fc` (custom), тот же, что делает `update.sh`; восстанавливается
  `pg_restore --clean --if-exists`.
- Файлы — `./backups/db-<UTC-таймстамп>.dump` (тот же каталог, куда пишет `update.sh`, —
  единая точка восстановления). Внутри контейнера каталог смонтирован как `/backups`.
- **Ротация** — `scripts/backup.sh` удаляет СВОИ дампы (`db-*.dump`) старше
  `BACKUP_RETENTION_DAYS` дней (по умолчанию 14; `0` = не удалять). Чужих файлов не трогает.
- Дампы содержат все данные (в т.ч. хеши паролей, TOTP-секреты) — скрипт ставит на файл
  `chmod 600`; каталог `./backups` держите `700`.

Параметры — в `.env.prod` (закомментированы в `.env.prod.example`, действуют дефолты):

```env
# BACKUP_RETENTION_DAYS=14      # сколько суток хранить дампы (0 = не удалять)
# BACKUP_INTERVAL_SECONDS=86400 # период авто-бэкапа (сутки)
```

Сервис поднимается вместе со стеком:

```bash
docker compose -f docker-compose.prod.yml up -d
docker compose -f docker-compose.prod.yml logs -f backup   # видно строки [backup …]
```

### Альтернатива — cron на хосте

Если не хотите держать сервис-цикл, вызывайте скрипт из cron хоста (клиент PostgreSQL
или тот же образ через `docker compose run`):

```cron
# ежедневно в 03:30 — дамп в ./backups с ротацией 14 дней
30 3 * * * cd /path/to/RoutineOps && BACKUP_DIR=./backups BACKUP_RETENTION_DAYS=14 \
  DATABASE_DSN="postgres://mdm:PASS@localhost:5432/mdm?sslmode=disable" scripts/backup.sh >> ./backups/backup.log 2>&1
```

`scripts/backup.sh` принимает `DATABASE_DSN` **или** стандартные `PG*`-переменные
(`PGHOST/PGPORT/PGUSER/PGPASSWORD/PGDATABASE`).

> Только дампа БД для полного DR **недостаточно**: отдельно храните `certs/`,
> `release_ed25519.pem`, `.env.prod` — см. [operations.md, раздел 2](operations.md).

## 6. Восстановление из дампа

`scripts/restore.sh` восстанавливает БД из дампа `backup.sh`/`update.sh`.
**Операция ДЕСТРУКТИВНА** (`pg_restore --clean --if-exists` дропает и пересоздаёт
объекты — текущие данные заменяются дампом), поэтому требует **явного подтверждения**:

```bash
# с хоста (клиент PostgreSQL):
DATABASE_DSN="postgres://mdm:PASS@localhost:5432/mdm?sslmode=disable" \
  scripts/restore.sh backups/db-20260723-030000.dump --yes

# или через контейнер postgres:
docker compose -f docker-compose.prod.yml exec -T -e BACKUP_RESTORE_CONFIRM=yes postgres \
  sh -c 'pg_restore --clean --if-exists -d "$DATABASE_DSN" -' < backups/db-20260723-030000.dump
```

Без флага `--yes` (или `BACKUP_RESTORE_CONFIRM=yes`) скрипт откажется работать —
защита от случайного затирания рабочей БД. После восстановления перезапустите сервер,
чтобы он работал поверх новых данных:

```bash
docker compose -f docker-compose.prod.yml restart server
```

Полная процедура «сервер умер, поднимаем с нуля из бэкап-набора» (код + секреты + БД +
перепубликация агентов) — [operations.md, раздел 3](operations.md).

## Границы и follow-up

Честно о том, чего пока НЕТ:

- **Авто-failover БД** — не входит в приложение. Отказоустойчивость Postgres/Redis
  обеспечивается стандартной репликацией или managed-сервисом (раздел 3). Сервер видит
  один `DATABASE_DSN`/`REDIS_ADDR`; переключение на реплику — задача того слоя.
- **Leader-election для фоновых тикеров** — follow-up. Сейчас при N узлах тикеры идут на
  каждом; чистки и реконсайлер от этого безопасны (идемпотентны / дедуп по `TaskID`), но
  детектор `agent_unreachable` может изредка задвоить алерт (раздел 4). Аккуратное решение —
  выбор лидера (advisory-lock в Postgres `pg_try_advisory_lock` или отдельный флаг «этот
  узел ведёт фон»), чтобы фон крутил ровно один узел. До этого — держите фон на одном узле
  либо смиритесь с редким дублем алерта.
- **Строгая уникальность алертов** — альтернатива leader-election для конкретно
  `agent_unreachable`: partial unique index, закрывающий гонку одновременных INSERT на
  уровне БД. Тоже follow-up (в коде отмечено как возможное усиление).

То, что **готово и безопасно эксплуатировать** сегодня: несколько stateless-узлов за LB
без sticky-сессий, `/readyz` для корректного вывода неготового узла из ротации,
регулярные авто-бэкапы с ротацией и предсказуемое восстановление.
