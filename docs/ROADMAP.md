# Роадмап RoutineOps

Актуально на 2026-07-22 (v2.4.5).

Этот файл — единственный источник правды по тому, что уже есть и что планируется.
Матрица «Free vs Enterprise» в [README.md](../README.md) — её краткая выжимка.

## Принцип границы

- **Free** — весь операционный MDM для одного администратора или небольшой команды:
  инвентарь, группы, скрипты и расписания, софт-политики, блокировка экрана, карантин,
  временные админ-права, алерты, аудит, self-update агентов. Free должен быть **мощным**:
  это то, ради чего продукт ставят.
- **Enterprise** — то, что нужно **организации**: корпоративная идентичность, изоляция
  между подразделениями, деструктивные операции с ключами восстановления, compliance-
  отчётность, масштаб и HA.
- **Базовая безопасность не продаётся.** mTLS, аудит, шифрование секретов, подпись
  релизов — во Free. «Paywall security» мы не делаем.

Enterprise-код физически отсутствует в Free-сборке (сборочный тег), а не «выключен
флагом»: выключенную проверку в открытом коде просто удаляют форком.

## Планируется (Enterprise)

### Этап 1 — текущий

| Возможность | Комментарий |
|---|---|
| ✅ Лицензионное ядро | **Реализовано (в main).** Подписанный офлайн-файл лицензии (ed25519) + argon2id-пароль активации, feature-gate (`mgr.Has`), срок и grace; деплойер = центр лицензирования (свой ключ, CLI `routineops-license`). Сборка `-tags enterprise` |
| FileVault: эскроу ключей восстановления и принудительная блокировка (macOS) | Серверная часть готова, агентская в работе |
| ✅ SSO / OIDC | **Реализовано (в main).** Вход через внешний OIDC-IdP (authorization code + PKCE S256), за лицензией `FeatureSSO`. Верификация подписи/iss/aud/exp через go-oidc (JWKS с ротацией) + явная сверка nonce; server-side single-use state/nonce/verifier (`sso_auth_flows`); ленивый discovery (IdP down на старте не роняет сервер). Матчинг по неизменяемой (issuer, sub), НЕ по email — SSO не линкуется к существующему аккаунту (email-коллизия → отказ), `auth_source='oidc'` блокирует password-login и forgot/reset. JIT-провижининг (unusable-пароль) + role-mapping из claim (allowlist, fail-closed, it_admin по умолчанию никогда). Кнопка «Войти через SSO» на логине, dev-Keycloak в `deploy/`. go-oidc/oauth2 только под `//go:build enterprise`. back-channel logout, ручной релинк — follow-up (SAML — отдельная строка ниже) |
| ✅ SAML 2.0 SSO | **Реализовано (в main).** SP-initiated SAML 2.0 рядом с OIDC, за той же лицензией `FeatureSSO`. AuthnRequest (HTTP-Redirect) + ACS с валидацией подписи/условий/audience/recipient/InResponseTo библиотекой `crewjam/saml` (своя XML-крипта не пишется). Провижининг по неизменяемой (IdP EntityID, NameID), отказ авто-линка при email-коллизии, JIT viewer, role-mapping из атрибута `groups` (allowlist, fail-closed, it_admin по умолчанию никогда). Публичные `/api/v1/auth/saml/{metadata,login,acs,status}` вне JWT-middleware; single-use `sso_auth_flows` (RelayState) + подписанный InResponseTo как anti-replay/CSRF. Конфиг из `SAML_*` env, `crewjam/saml`+goxmldsig только под `//go:build enterprise` (0 в free-сборке). Миграция 058 (partial-unique по (issuer,sub) WHERE auth_source='saml'). SLO/IdP-initiated/encrypted assertions — follow-up |
| ✅ MFA / TOTP | **Реализовано (в main).** Двухшаговый вход: пароль → TOTP (RFC 6238, SHA1/6/30, совместимо с Google Authenticator/Authy/1Password). Секрет шифруется AES-256-GCM ключом `ROUTINEOPS_MFA_ENC_KEY` (env, не в БД, AAD=user_id); challenge между шагами — одноразовый DB-backed токен (не сессия). Recovery-коды (120 бит, sha256, одноразовые) + admin-reset залоченному. Replay-guard по last_step, per-account backoff на оба шага. **БАЗОВАЯ фича (не за лицензией)** — включается в профиле любым юзером. Потеря enc-ключа не блокирует вход (recovery/admin-reset). **Enforce-by-policy реализовано (в main):** обязать MFA для роли (`org_settings`, миграция 054) — гейт `requireMFAEnrollment` с anti-lockout allowlist (`/me`, enroll, confirm, logout всегда доступны — залоченный включит MFA и разблокируется), сервисные токены пропускаются, сигнал `mfa_required` в веб (баннер + редирект на `/profile`) |
| ✅ SCIM-провижининг | **Реализовано (в main).** SCIM 2.0 `/scim/v2/Users` (list с фильтром/пагинацией, create, get, PUT/PATCH, DELETE=деактивация) для авто-синхронизации юзеров из IdP (Okta/Azure AD/OneLogin). Публичный канал за СОБСТВЕННЫМ bearer-токеном (не JWT; хранится sha256, ротация админом, показ один раз) и лицензией (`FeatureSCIM`). JIT-юзеры `auth_source='scim'` (unusable-пароль), деактивация (`is_active=false`) гейтит и живую сессию, и логин; `auth_source != 'local'` не даёт локального входа/reset в обход IdP. **Маппинг SCIM-групп на роли реализован (в main):** таблица `scim_role_mapping` (миграция 056), при провижининге группы IdP разрешаются в роль (allowlist, fail-closed, it_admin по умолчанию никогда) |
| ✅ **Удаление ПО с устройства из интерфейса** | **Реализовано (в main).** Кнопка «удалить» в списке ПО карточки → тихая деинсталляция задачей агенту (MSI/QuietUninstallString/Inno) + подтверждение + аудит; за лицензией (`FeatureSoftwareRemoval`), гейтится в вебе через `/capabilities` |
| ✅ **Мультитенантность (MVP)** | **Реализовано (в main).** Модель тенантов (`tenants`, неудаляемый «Default» с фиксированным id) + привязка `tenant_id` на `devices`/`users` (nullable + DEFAULT=Default, `ON DELETE SET NULL` — существующие сущности бэкфилены, ничего не сломано). CRUD тенантов + назначение устройств/юзеров (`POST /tenants/{id}/assign`), счётчики, guard'ы (default неудаляем, непустой → 409, валидация slug), за лицензией (`FeatureMultitenancy`), страница «Тенанты». **ПОЛНАЯ per-query изоляция данных РЕАЛИЗОВАНА (в main):** ctx-шов (`storage.WithTenantScope`, `jwtMiddleware` резолвит тенант актора per-request) + предикат `($k IS NULL OR col = $k)` на ~70 storage-методах (устройства/задачи/алерты/телеметрия/help/access-requests/пользователи/аудит/compliance/отчёты/CVE/RD + device-сторона групп). Default-тенант = провайдер (нескоуплено, видит всё → single-org без изменений); не-Default изолирован (кросс-тенант читается пусто, мутация по id → 404). Deployment-shared (follow-up Phase B): библиотека скриптов, script-политики, метаданные групп, ростер миграции, токены энроллмента/инвайтов; нотификация алертов всем админам. Изоляция и модель — [multitenancy-scoping.md](multitenancy-scoping.md), тесты изоляции storage+API |
| ✅ Экспорт аудита в SIEM | **Реализовано (в main).** Фоновый форвардинг событий аудита на webhook с HMAC-подписью, durable-курсор (safety-lag), за лицензией (`FeatureSIEMExport`), страница «SIEM-экспорт». syslog / CEF — follow-up |
| ✅ Tamper-evident аудит | **Реализовано (в main).** Кейд хеш-цепочка над `audit_log` (`entry_hmac = HMAC(key, prev‖canonical)`, ключ в env), подпись включается автоматически при заданном `ROUTINEOPS_AUDIT_HMAC_KEY` — ловит модификацию/удаление середины/вставку/replay (для подделки нужен ключ, которого нет в БД); голова цепочки (`audit_chain`) — якорь против наивного усечения хвоста. Проверка `GET /audit-log/verify` за лицензией (`FeatureAuditIntegrity`), кнопка на странице «Аудит». Полная защита от усечения хвоста — внешний якорь (SIEM-экспорт) |
| ✅ Compliance-дашборды и отчёты | **Реализовано (в main).** `GET /compliance/report` — общий скор (0–100) + ~10 проверок SQL-агрегатами по существующим данным: SOC2 (админы с MFA, adoption 2FA, устаревшие admin-access-заявки), CIS (шифрование диска), access (check-in устройств, backlog энроллмента), inventory (владельцы, software/script-политики), audit (tamper-evidence сконфигурирован). CSV-экспорт, за лицензией (`FeatureCompliance`), страница «Соответствие». Без миграции — только агрегация |
| ✅ Уровни критичности алертов и маршрутизация | **Реализовано (в main).** `severity` (info/warning/critical) на алертах + правила `alert_routing_rules` (min_severity → канал telegram/webhook + target). Фоновый роутер-поллер (по образцу SIEM-экспортёра, durable-курсор): доставка алертов `severity >= min_severity` по включённым правилам + эскалация неотвеченных `critical` старше порога. **Per-(alert,rule) состояние эскалации (в main):** отдельная таблица `alert_rule_escalations` (миграция 057, атомарный upsert `TryEscalateRule`) вместо единой колонки last_escalated_at — устраняет MVP-баг, когда быстрое правило сбрасывало анти-спам-курсор медленному. За лицензией (`FeatureAlertRouting`), severity-бейджи, страница правил. Дежурства/расписания — follow-up |
| ✅ Сканирование ПО на уязвимости (CVE) | **Реализовано (в main).** Матчинг инвентаря `device_software` против фида уязвимостей: деплойер заливает выгрузку NVD/OSV (`POST /cve/feed` — JSON-массив, замена целиком; `DELETE /cve/feed`), `POST /cve/scan` пересобирает находки одним SQL-JOIN'ом (регистронезависимая подстрока имени + version-логика в Go: `<`/`<=`/`=`/`>`/`>=`, покомпонентное сравнение), `GET /cve/findings`/`/cve/summary`. Pending-устройства исключены, аудит `cve_feed_loaded`/`cve_scan_run`, за лицензией (`FeatureCVEScan`), страница «Уязвимости». **Внешний сетевой фид реализован (в main):** конфиг источника `cve_feed_source` (миграция 055) + фоновый `cvesync`-синкер — по расписанию тянет фид с настроенного URL, заменяет его и (опц.) пересобирает находки; форс-синк из UI. Точный CPE-матчинг — follow-up |
| ✅ HA / несколько узлов, авто-бэкапы | **Частично реализовано (в main).** `/readyz` (реальная проба БД/Redis — для LB/оркестратора не слать трафик на неготовый узел; `/healthz` = liveness). Авто-бэкапы Postgres: `scripts/backup.sh` (pg_dump -Fc + ротация) + `restore.sh`, compose-сервис `backup`. Сервер stateless (сессии в JWT) → N узлов за LB без sticky-сессий (`docs/ha-and-backups.md`). **Follow-up:** leader-election фоновых тикеров при N узлах (чистки идемпотентны, но детектор `agent_unreachable` может задвоить алерт), авто-failover, репликация Postgres — деплойерские/managed |
| ✅ Policy-as-code / GitOps | **Реализовано (в main).** Декларативные глобальные software-правила как JSON source-of-truth (`policy_declaration`, миграция 051): админ хранит желаемый набор правил, сервер реконсилит живое состояние и показывает дрейф (что в декларации, но не применено, и наоборот). За лицензией (`FeaturePolicyAsCode`), страница «Policy-as-code». Прямой git-pull источника — follow-up |
| ✅ Авто-устранение запрещённого ПО | **Реализовано (в main).** Фоновый ремедиатор (`FeatureAutoRemediation`, миграция 052): периодически ищет forbidden-нарушения инвентаря и, если авто-устранение включено конфигом (**дефолт — ВЫКЛ, деструктивно**), ставит `remove_software`-задачи, переиспользуя проверенный путь удаления ПО. Настройка + лог в UI, страница «Авто-устранение ПО» |
| ✅ Отчёты / экспорт CSV | **Реализовано (в main).** `FeatureReports`: агрегатные отчёты по существующим данным (устройства, инвентарь ПО, комплаенс, алерты) + CSV-экспорт, без миграции. Печатный HTML→PDF — follow-up |

## Как это устроено технически

- **Обновление сервера** — `git pull` + пересборка контейнеров; миграции накатывает
  отдельный сервис до старта сервера (fail-closed). См.
  [self-hosted-deploy.md](self-hosted-deploy.md).
- **Обновление агентов** — парк тянет подписанный бинарь с сервера **своего** деплойера.
  Мы не пушим ничего в чужую инфраструктуру.
- **Enterprise-доставка** — готовый образ сервера из приватного реестра по токену
  подписки; исходники Enterprise не отдаются. Агент у Enterprise-клиента — тот же
  публичный Free-агент, кроме FileVault-сценария.

Вопросы по Enterprise и предложения по роадмапу — через Issues репозитория.
