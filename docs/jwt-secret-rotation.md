# Рунбук: ротация JWT_SECRET (прод)

Актуально на 2026-07-15 (v2.4.3).

`JWT_SECRET` — единственный корень доверия для admin-сессий (симметричный HS256).
Кто прочитал секрет — бесконечно форжит admin-токены; `jti`-блоклист против свежих
`jti` бесполезен. Единственный стоп-кран при компрометации — ротация (разлогинит всех).

## Когда ротировать
- Подозрение на утечку: доступ к env контейнера / `.env.prod` / бэкапам / дампу процесса.
- Планово раз в N месяцев.
- После ухода человека, имевшего доступ к проду.

## Требования к новому секрету
- ≥32 байта, высокая энтропия. Стартовый guard (`cmd/server/main.go`) отвергает короткие
  (<32) и низкоэнтропийные (<16 различных байт) секреты — сервер не поднимется на слабом.
- Генерация: `openssl rand -base64 48` (не печатать в общие логи/чат).

## Процедура (прод)
1. `NEW=$(openssl rand -base64 48)` — сгенерировать, никуда не эхоить.
2. Обновить ключ `JWT_SECRET` в `.env.prod` (или env сервиса) на `$NEW`. Права файла → `chmod 600`.
3. Рестарт сервера: `docker compose -f docker-compose.prod.yml up -d server`
   (или рестарт systemd-сервиса).
4. Последствие: ВСЕ активные сессии инвалидируются (старые токены не проходят проверку
   подписи) → админы логинятся заново. `token_blocklist` чистить не нужно.
5. Проверка: старый (до ротации) токен → 401; свежий логин → 200.

## Смежное
- TTL токена = 8ч (`internal/server/api/auth.go`) — даже без ротации украденный токен живёт ≤8ч.
- Смена/сброс пароля ОТЗЫВАЕТ все ранее выпущенные токены пользователя (token-epoch,
  миграция 024; владельцу при смене собственного пароля кука переминчивается). Logout
  ревокирует токен по `jti`. Против утечки самого СЕКРЕТА это не помогает — атакующий
  минтит новые токены; единственный стоп-кран — ротация.
- Секрет не должен попадать в git — держать в `.env.prod` (в `.gitignore`), не в конфиге репо.
