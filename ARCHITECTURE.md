# Архитектура

Обзор для оператора: что ставится на сервер, как компоненты связаны и что от этого ожидать в эксплуатации. Версия продукта — в файле `VERSION`. Установка — [docs/install.md](docs/install.md) и [docs/self-hosted-deploy.md](docs/self-hosted-deploy.md).

## 1. Компоненты

Сервер — один Go-бинарь (монолит), развёртывание — Docker Compose. Агенты ставятся на устройства парка (Windows — MSI, macOS — .pkg, Linux — systemd-юнит через сгенерированный installer-скрипт) и работают как системная служба.

```
 Устройства парка                        Ваш сервер (docker compose)
┌──────────────────┐                    ┌─────────────────────────────────────┐
│  Агент Windows   │                    │  nginx (web)                        │
│  Агент macOS     │                    │   80  → редирект на 443             │
│  Агент Linux     │                    │         (кроме /downloads/)         │
└────────┬─────────┘                    │  443  → веб-UI + REST API,          │
         │                             │         enrollment, загрузки        │
         │ gRPC + mTLS ──────────────────► server (Go, монолит)              │
         │ :50051                       │  50051 → gRPC агентов (mTLS)        │
         │                             │  8081  → REST API мимо nginx        │
         │ HTTPS (enrollment,           │     │            │                  │
         │ self-update) ────────────────►    │            │                  │
                                        │     ▼            ▼                  │
 Администратор                          │  PostgreSQL   Redis (asynq —       │
┌──────────────────┐   HTTPS :443       │  (данные)     очередь задач)       │
│  Браузер (React) ├────────────────────►                                    │
└──────────────────┘                    └─────────────────────────────────────┘
```

Порты:

| Порт | Наружу | Назначение |
|---|---|---|
| 80 | да | редирект на 443; исключение — `/downloads/` (бинари агентов публичны, установщик сверяет sha256-пин) |
| 443 | да | веб-интерфейс, REST API, enrollment, `/ca.crt`, загрузка агентов |
| 50051 | да | постоянный gRPC-канал агентов (TLS, клиентские сертификаты) |
| 8081 | нет — bind 127.0.0.1 | REST API напрямую, минуя nginx. `/healthz` доступен только здесь — nginx его не проксирует |
| 5432 (PostgreSQL) | нет — bind 127.0.0.1 | данные |
| Redis | нет — только внутри compose-сети | очередь доставки задач |

Порты и переменные окружения подробно — [docs/install.md](docs/install.md).

Долговременное состояние живёт в PostgreSQL (named volume `postgres_data`) и в каталогах хоста `certs/` и `releases/`. Redis — только очередь доставки; источник правды по желаемому состоянию устройств — PostgreSQL.

## 2. Идентичность устройств

Каждый агент имеет клиентский mTLS-сертификат, выданный приватным CA сервера. `device_id` устройства — это CN сертификата, и только он: сервер не доверяет идентификаторам в теле сообщений. Подменить «своё» устройство на чужое без чужого ключа нельзя.

Регистрация (enrollment): администратор в UI создаёт устройство и получает одноразовый токен (TTL 24 часа, single-use). Агент генерирует ключ локально, отправляет CSR на `POST /api/v1/enroll` с этим токеном, сервер подписывает сертификат и сам проставляет CN. Приватный ключ устройства сервер не видит. Подробно — [docs/enrollment.md](docs/enrollment.md).

## 3. Потоки данных

- **Heartbeat** — лёгкий, частый (по умолчанию ~30 с) сигнал «жив» по постоянному gRPC-стриму. По этому же стриму сервер доставляет агенту задачи и команды (запуск скрипта, блокировка и т.д.); доставка идёт через очередь asynq с повторами.
- **Инвентаризация** — отдельный, более редкий (по умолчанию раз в 5 минут) и полный отчёт (ОС, железо, серийный номер, установленное ПО, версия агента). Разнесение heartbeat и инвентаризации держит постоянный канал дешёвым при любом размере парка.
- **Результаты скриптов** идемпотентны: каждый запуск имеет `run_id`, повторная отправка того же результата (после обрыва связи) не создаёт дубликатов.
- Устройство без heartbeat дольше порога `AGENT_UNREACHABLE_MINUTES` порождает алерт `agent_unreachable` (с подавлением дребезга от сна/modern standby).

## 4. Автономность агента

Агент не требует постоянной связи с сервером. Скрипт-политики с расписанием исполняются локальным планировщиком (cron-выражения) — по расписанию, даже когда сервер недоступен. В офлайне агент продолжает работать: политики применяются, блокировка экрана держится (офлайн-разблокировка по локально хранимому bcrypt-хешу пароля), результаты и события досылаются при восстановлении связи. При реконнекте агент синхронизирует политики и desired-состояние (например, блокировку) с сервером.

## 5. Обновления

**Агенты** обновляются сами: периодически (по умолчанию раз в 6 часов) запрашивают манифест релиза, сверяют sha256 и Ed25519-подпись манифеста публичным ключом релиза (агент получает его при enroll по доверенному каналу; сборки по умолчанию универсальные, без вшитого ключа — без валидной подписи обновление не применяется), атомарно заменяют себя и перезапускаются. Даунгрейд невозможен (anti-rollback floor); битый релиз чинится только версией вперёд. Приватный ключ подписи (`release_ed25519.pem`) генерируется при установке и уникален для вашей инсталляции. Его потеря **не** оставляет парк навсегда без обновлений: новый ключ раздаётся через `RELEASE_PUBKEY` в `.env.prod` и переэнроллмент — уже развёрнутые устройства подхватят его при переустановке/переэнроллменте, а бинарь остаётся универсальным (подробно — [docs/operations.md](docs/operations.md)). Подробно — [docs/self-update.md](docs/self-update.md).

**Сервер** обновляется скриптом `update.sh`: бэкап БД → `git pull` → пересборка. Миграции схемы накатывает отдельный fail-closed migrate-сервис **до** старта сервера: если миграция не прошла, сервер не стартует на несовместимой схеме. Down-миграций нет — откат только из бэкапа, который `update.sh` делает автоматически перед каждым обновлением.

## 6. Модель безопасности

Коротко: mTLS на всём канале агентов (сервер требует клиентский сертификат, идентичность = сертификат); веб — HTTPS с httpOnly-cookie-сессиями (JWT, TTL 8 ч, инвалидация при смене пароля) и RBAC на две роли (`it_admin` — всё, `viewer` — только чтение, проверка на сервере, не только в UI); все мутации пишутся в журнал аудита с настраиваемым retention. Подробности, границы модели угроз и процесс сообщения об уязвимостях — [SECURITY.md](SECURITY.md).

## 7. Open-core

Free-сборка (то, что собирает этот репозиторий по умолчанию) — полный операционный MDM: инвентаризация, скрипты (разовые и по расписанию), группы, политики ПО, алерты, блокировка устройств, аудит, RBAC, self-update. Enterprise-функции собираются отдельным build-tag и в Free-сборку не входят — остаются заглушки: соответствующие API отвечают Unimplemented/409. Free не деградирует без enterprise — это независимо работающий продукт, а не демо.
